## Scope and dependency policy

### Goals

- **Modern SpikeInterface API**: All v2 paths use `SortingAnalyzer` (SI ‚â•0.101), `PreprocessingPipeline` (SI ‚â•0.103), and current `spikeinterface.curation` primitives.
- **Cross-session unit tracking**: Pluggable matcher backend with UnitMatchPy as the first plugin. Schema (`TrackedUnit`) treats a "biological unit observed across N sessions" as a first-class queryable entity. DeepUnitMatch swappable via the same plugin slot (future).
- **Chronic / large recording support**: Session groups, materialized concatenated recording caches (same-day default; multi-day opt-in behind `allow_multi_day=True` with an explicit motion-correction preset ‚Äî see resolved decision #4 below), lazy streaming, sparse SortingAnalyzers, motion correction (rigid_fast default; DREDge available as an opt-in preset). 30 kHz √ó 128 ch √ó 24 h recordings (current Frank-lab polymer-probe scale; see [PMC11463554](https://pmc.ncbi.nlm.nih.gov/articles/PMC11463554/)) produce one canonical NWB-resident `Recording` artifact per recording, and concat workflows add one motion-corrected, unwhitened `ConcatenatedRecording` cache per selected group/parameter tuple (see [shared-contracts.md ¬ß Recording Cache Format](shared-contracts.md#recording-cache-format)). For cross-day analyses the recommended path is Phase 4 sort-then-match (UnitMatch), not multi-day concat.
- **First-class unit ‚Üí brain region traceability**: every curated unit row carries (at minimum) the peak channel and the brain region(s) for that channel, persisted at sort time and resolvable via a single query on `CurationV2` or `SpikeSortingOutput`. v1's `CurationV1.get_sort_group_info` samples *one* electrode per sort group; that's wrong for multi-region polymer probes. v2 walks every electrode in the sort group, persists per-unit peak-channel and region, and exposes `Sorting.get_unit_brain_regions(key) -> pd.DataFrame` and the same accessor on `CurationV2`. Brain region tracing must work end-to-end through `SpikeSortingOutput.get_sort_group_info` and through `TrackedUnit` (per-session brain regions for matched units).
- **Reduced UX friction**: `run_v2_pipeline(nwb_file_name, sort_group_id, interval_list_name, team_name, pipeline_preset=...)` runs the standard single-session flow and returns the final `merge_id` plus a manifest; `list_pipeline_presets()`/`describe_pipeline_presets()` expose the preset catalog and `preflight_v2_pipeline()` validates setup before populate. *(Planned, not yet shipped:* a concatenated sort-and-curate flow ‚Äî Phase 3 `ConcatenatedRecording` ‚Äî and a separate `run_v2_unit_match(session_group_owner, session_group_name, curation_choices=...)` cross-session sort-then-match helper ‚Äî Phase 4 ‚Äî so the concat path is not overloaded.) Pydantic models validate parameter dicts at insert time. Notebook walkthrough drops from 35 cells to ~8.
- **Drop-in for existing downstream consumers**: New `SpikeSortingOutput.CurationV2` part. Decoding, ripple, MUA, `SortedSpikesGroup` work unchanged.
- **Explicit v1 parity boundaries**: [feature-parity.md](feature-parity.md) is the contract for which v1 workflows are preserved, which are replaced by successor tables, and which departures are intentional.
- **Team-based sorting preserved**. v2 retains v1's `LabTeam` FK structure: `RecordingSelection`, `SessionGroup.Member`, and `run_v2_pipeline(team_name=...)` all carry team ownership. `cautious_delete` continues to check team membership; per-team data partitioning, shared-team sorting, and team-based access control all work the same way they did in v1. Two teams running an identical sort get distinct `recording_id` UUIDs and distinct `AnalysisNwbfile` rows ‚Äî preventing the v0 filesystem-overwrite bug ([#133](https://github.com/LorenFrankLab/spyglass/issues/133)).
- **Pre-production schema policy** (updated from the earlier hard zero-migration rule): v2 is still pre-production, so project-owner-approved table definition edits may be made in place, with no migration/deprecation period, when a plan explicitly lists the changed tables and validation. The review-fixes checkpoint remains the historical baseline; later authorized corrections include tracked analyzer waveform params, sorter execution params, metric waveform identity, and surfaced template metric columns. Once v2 reaches production/freeze, new table-shape changes require an explicit migration plan rather than direct def edits.
- **v0/v1 coexistence with an explicit two-environment regime**: Legacy source and existing rows remain in-tree/queryable where possible throughout the v2 lifetime. Under the SI 0.104 environment, v2 is the supported runtime path for new processing; active v0/v1 runtime workflows require the legacy SI 0.99 environment unless Phase 0c explicitly ports a narrow compatibility shim. This is the largest user-facing operational consequence of the plan: until v2 reaches parity, labs that still re-run active v0/v1 spike-sorting workflows must maintain a legacy environment alongside the SI 0.104 v2 environment. **No v1 source removal is planned in this work**.

### Non-Goals

- **Re-sorting existing v1 data.** v2 adds the new pipeline; users may run it on new sessions but the plan never invalidates or migrates existing v1 sorts.
- **A new merge table.** v2 plugs into the existing `SpikeSortingOutput`; adding `MergeV2Output` would force downstream code to choose between merges. Not in scope.
- **A v2-specific imported-sorting table.** Imported external Units remain in the existing `ImportedSpikeSorting` table and merge part. v2 compares against or coexists with imported Units; it does not absorb them into `CurationV2` lineage.
- **Removing v0 or v1 source.** Both stay in `src/spyglass/spikesorting/{v0,v1}/`. Documentation marks active legacy runtime workflows as requiring the legacy SI 0.99 environment unless explicitly ported. **No v1 source-removal sunset is planned here.**
- **DeepUnitMatch.** Pluggable via the Phase 4 matcher protocol but not implemented; future work.
- **Custom UI / web app.** All curation UI uses upstream FigPack; we do not build a Spyglass-specific viewer.
- **Cross-region / cross-probe matching.** UnitMatch assumes a single probe across sessions in a session group. Multi-probe handling is out of scope.

## Metrics

How we'll know v2 is working:

- **Ground-truth tests (Phase 1)**: Run v2 against MEArec-generated polymer-probe and Neuropixels fixtures with known planted units; compute precision/recall via `spikeinterface.comparison.compare_sorter_to_ground_truth`. The 128-channel LLNL polymer probe (current Frank-lab implant; [PMC11463554](https://pmc.ncbi.nlm.nih.gov/articles/PMC11463554/), [Neuron 2026 S0896-6273(26)00048-6](https://www.cell.com/neuron/fulltext/S0896-6273(26)00048-6)) is the primary lab-relevant fixture and must use a real SpikeInterface sorter path such as MS5/KS4 for the correctness gate. `clusterless_thresholder` and minirec serve only as plumbing / deterministic parity guards; minirec is too short to contain real spikes. v1 parity moves to the env-var real-data path (`SPIKESORTING_V2_REAL_NWB_PATH`); no minirec parity test ships.
- **Database safety**: DataJoint integration validation runs against the isolated pytest Docker MySQL path or a manual Docker/database-prefix equivalent. Production-connected real-data checks are optional smoke tests gated by `SPYGLASS_ALLOW_PRODUCTION_SMOKE=1`, write only to test schemas/temp output paths, and never exercise destructive cleanup on production rows.
- **Memory: streamed `Recording` write (Phase 1b)**: `Recording.populate()` on a synthetic 5-minute 32-channel 30 kHz fixture must stay under ~15 GB peak RSS (‚âà`2 √ó buffer_gb √ó headroom`); without the chunked-iterator port, the same write needs the full preprocessed float64 array (~18 GB) in RAM and OOMs on most workstations. Smoke test is the regression gate.
- **Memory and runtime (Phase 3)**: Single-channel-group, 30 kHz, 1 h recording must complete `Sorting.populate()` in <10 min and peak RSS <8 GB on a 16-core machine with default job kwargs. Smoke-test on 5 min slice first. Phase 1b's chunked write and `_parallel_make = True` are prerequisites: without streaming the preprocessed `ElectricalSeries`, the upstream `Recording.populate()` step alone exceeds this RSS target.
- **Notebook cells (Phase 5)**: The end-to-end notebook for a single sort drops from 35 code cells to ‚â§10. Counted manually before merge.
- **Downstream non-regression (Phase 1)**: A `SortedSpikesGroup` built from a v2 `CurationV2` row produces the same `get_spike_times` shape and units as legacy merge rows. Legacy comparison fixtures are captured from the SI 0.99 environment unless Phase 0c explicitly ports the relevant v1 runtime surface.
- **UnitMatch polymer validation gate (Phase 4)**: On the MEArec-generated 2-session polymer fixture (Phase 4b task; planted shared templates across the 128-channel LLNL polymer probe ‚Äî 4 shanks √ó 32 ch), `test_v2_unitmatch_polymer_mearec_ground_truth` asserts AUC > 0.85 of match probability against ground-truth template correspondence. Polymer is the gate because it is the current Frank-lab probe. Supplementary Neuropixels, tetrode, or real-data validation belongs in follow-up work.

## Risks and Mitigations

| Risk | Mitigation |
| --- | --- |
| Sorter output stochasticity makes parity testing noisy. | Phase 0 captures v1 baseline with `clusterless_thresholder` (deterministic peak-detection path; v1 parameters do not include a seed) as the tight spike-time parity test. **MS4 is not deterministic**, and KS4 is a PyTorch CPU/GPU sorter whose output should not be treated as a portable exact-output oracle even though Kilosort seeds NumPy/PyTorch internally. MS4/MS5/KS4 must never be used for exact rerun or exact v1-v2 parity gates; comparisons are bounded by qualitative metrics (unit count order, firing-rate distribution shape) and explicitly noted. |
| SortingAnalyzer folders are large for chronic recordings (50+ GB possible). | Phase 1 stores analyzer folder paths in DataJoint, **not** content. Phase 2 ports the v1 `RecordingRecompute` pattern into v2-specific `RecordingArtifactRecompute*` and `SortingAnalyzerRecompute*` tables for verified storage reclamation. Sparse=True default cuts size 5-10√ó. |
| Real-data validation could accidentally write to or delete from the production database/storage. | Real-data tests are not the default validation target. The default integration target is the isolated Docker/test-prefix database with temporary Spyglass directories. Production-connected smoke requires `SPYGLASS_ALLOW_PRODUCTION_SMOKE=1`, is read-mostly, and must not call destructive cleanup against production rows or analysis storage. |
| UnitMatch's published validation is Neuropixels-heavy and may not transfer to Frank-lab polymer/tetrode data. | Phase 4 gates shipping on a MEArec polymer-probe fixture, because polymer is the lab-relevant standard for this workflow. Neuropixels and tetrode AUCs are recorded as informational checks; tetrode users are not promised production matching unless the informational result supports it. |
| Adding a 4th part to `SpikeSortingOutput` may surface latent merge bugs. | Phase 1 includes integration tests against `SortedSpikesGroup`, decoding, ripple, MUA with a v2-sourced merge_id. |
| `set_group_by_shank()` from v1 silently overwrites sort groups ‚Äî v2 must not perpetuate this. | `SortGroupV2.set_group_by_shank()` raises if rows already exist for the session; destructive replacement requires `delete_existing_entries=True, confirm=True` after reviewing a `DeletionPreview`. Phase 1 task. |
| Pydantic schema migration risk: existing parameter blobs may not validate. | All Pydantic models are introduced fresh in v2 Lookup tables; v1 Lookup blobs remain unvalidated. No migration of v1 params. |
| Concatenation across days (out of scope but tempting). | `SessionGroup` derives member dates from `Session.session_start_time`; `ConcatenatedRecording.populate()` raises if members span multiple dates without an explicit `allow_multi_day=True` parameter row. |
| Pipeline helpers may hide reproducibility. | `run_v2_pipeline()` returns a manifest dict of every `(table, key)` it touched; notebooks print it. The planned Phase 4 `run_v2_unit_match()` will do the same. |
| The SI 0.104 global pin bump breaks non-spike-sorting consumers. | Phase 0c audits every `spikeinterface` import outside `spikesorting/` before changing the project pin, including waveform utilities, analysis mixins/builders, `common_nwbfile`, and decoding waveform-feature code. It classifies each as query-compatible, guarded legacy-only, or safe-to-port, and adds tests or explicit guards. |
| Future `make()` refactors silently drop the chunked-iterator path or `_parallel_make` flag, reintroducing the Phase 1 OOM and serialized-populate regressions. | Phase 1b's `test_make_compute_is_pure` static check guards against tri-part collapse; the `test_recording_peak_rss_under_threshold` smoke test guards against re-introducing `recording.get_traces(...)` at the top level of `_write_nwb_artifact`. Phase 2's `RecordingArtifactRecompute*` write path must reuse the streaming helper from `spyglass.spikesorting.v2._nwb_iterators`, not roll a parallel materialization path. |

## Rollout Strategy

v2 is **strictly additive** for the lifetime of this plan. No feature flags, no parallel-running shims, no deprecation phase needed inside this plan.

- v1 stays as the documented production path for the legacy SI 0.99 environment until v2 reaches parity (defined as: Phases 1‚Äì5 merged, parity tests green on lab production data for one cycle).
- Under the SI 0.104 environment, v2 is the documented runtime path for new spike sorting from Phase 1 onward; docs keep legacy-env instructions for active v0/v1 processing.
- After parity is achieved, the documentation root (`docs/src/Features/`) gets a banner pointing users to v2 for new work; v1 docs remain with legacy-environment guidance.
- **v1 source code removal is explicitly out of scope** for this plan. Phase 5 documents how users choose v1 vs v2; it does not define a deletion or sunset trigger.
- Execution uses the long-lived `spikesorting-v2` integration branch with checkpoint commits. Phases are implementation checkpoints, not required standalone PRs. The project owner chooses PR boundaries; a PR may group multiple checkpoints if their order, gating evidence, and review surface remain clear. Phase 1 alone is usable for opt-in SI 0.104 v2 single-session processing, but the lab-wide recommended production default remains v1 until the parity criterion above is met.

## Resolved Design Decisions

These were open questions; user has decided. Documented here so subsequent phase implementers don't re-litigate.

1. **MountainSort 4 stays in v2, but not as a deterministic oracle.** Add `mountainsort4` to `SorterParameters` defaults alongside `mountainsort5`, `kilosort4`, `spykingcircus2`, `tridesclous2`, and the Spyglass special-case `clusterless_thresholder` peak-detection path. MS4 installs on Linux and the v1 sorter_params dicts can be ported with minimal changes (notably removing the `tempdir` field-mutation hack). Pydantic schema for MS4 ships in Phase 1's `_params/sorter.py`; non-Linux resolver/runtime gaps get platform-specific guards rather than removing MS4 globally. Tests may assert that MS4 runs and produces sane bounded outputs, but not exact unit IDs, exact spike times, or exact rerun equality.
2. **FigPack is the v2 curation UI, gated by a Phase 5a feasibility spike.** FigURL is not extended to v2; FigURL stays available for v1 data only. Phase 5a must verify the installable packages, view-construction API, upload/publish API, and edited-curation state round trip before any FigPack DataJoint table implementation begins. If FigPack is genuinely unusable, the plan stops there and surfaces the blocker to the project owner rather than silently shipping a degraded fallback.
3. **DeepUnitMatch deferred.** Phase 4's `MatcherProtocol` slot accepts it as a future plugin; not implemented in this plan.
4. **Multi-day concatenation is supported by the schema, but is opt-in and not the recommended cross-day path.** `SessionGroup.create_group(..., allow_multi_day=True)` is required to insert members spanning ‚â•2 dates; the default rejects multi-day inputs with a clear pointer to Phase 4 sort-then-match. `ConcatenatedRecording` does NOT auto-dispatch DREDge for multi-day groups ‚Äî the caller picks an explicit motion-correction preset. **Sort-then-match (Phase 4 UnitMatch) is the recommended cross-day workflow, and Phase 4's gating validation is the Frank-lab polymer-probe fixture.** Neuropixels, tetrode, and real-data UnitMatch checks are useful follow-up validation, not Phase 4 gates. Multi-day tetrode sorting is not a current Frank-lab use case, so neither multi-day tetrode concat nor tetrode UnitMatch is treated as a gating workflow. Both remain schema-supported.
5. **Schema policy is pre-production-direct-edit, not production migration.** The earlier hard zero-migration rule is superseded while v2 remains pre-production. Direct table definition edits are allowed only when the project owner authorizes them and the owning plan documents the affected tables, provenance reason, and tests. The post-review-fixes baseline still explains where the current schema came from, but it is no longer a blanket ban on later pre-production corrections. Once v2 reaches production/freeze, this permissive policy ends and table-shape changes need explicit migration planning.
6. **Motion estimates are not persisted in the MVP concat schema.** Phase 0c verifies the pinned SI `correct_motion` API before Phase 1 freezes `MotionCorrectionParameters` and `ConcatenatedRecording`, but the MVP deliberately persists only the corrected, unwhitened `ElectricalSeries`, sample boundaries, and content hash. Motion trajectories / motion-info side artifacts are not queryable in Phase 3. A future measured need can add a separate opt-in motion-output table; it must not alter the frozen `ConcatenatedRecording` row shape.
7. **Phase 1b added `CurationV2.MergeGroup` as an early schema correction.** A systematic v1‚Üív2 audit during Phase 1b development found that v2's `_build_curated_unit_rows` bakes merge groups into `CurationV2.Unit`'s "kept unit" set with the contributor list NOT persisted anywhere queryable, breaking v1's ability to recover "which units were merged into which" after `insert_curation`. v1's pattern stores merge_groups as a NWB unit column. The user chose a part table over the v1 NWB-column pattern for queryability and FK validation: bulk-audit queries ("show me every pair (kept_unit, contributor) across sortings"), provenance ("for this paper, list every merge decision"), and DataJoint-level FK enforcement that contributor unit_ids reference real `Sorting.Unit` rows are easier with a part table. This is now one historical example of the pre-production direct-edit policy in decision #5, not a unique exception under a frozen schema. The MergeGroup table's exact schema is defined in [phase-1b-runtime-regressions.md ¬ß Tasks (B3)](phase-1b-runtime-regressions.md); the review-fixes corrections are captured in the post-review-fixes schema baseline.

8. **Phase 1b absorbs the full v1‚Üîv2 parity audit (four sweeps, 50+ items).** Phase 1b was originally scoped to two runtime regressions (memory + tri-part `make`). Over the course of plan development, four independent audit passes (single-agent pass 1, single-agent pass 2, then an 8-parallel-agent deep sweep) surfaced 18 + 5 + 30 = 53 net additional items where v2 silently changed v1's behavior. The user's governing principle for the expanded scope: **"don't change v1 behavior without a documented reason."** Every v2 divergence either:
   - matches v1 (default ‚Äî applies to 42 of the R/N items),
   - is preserved because v1's behavior is a bug (B7 + two N-items: v1 `amplitude_thresh_uV` unit fix, heterogeneous-gain raise ‚Äî see decision #9 below),
   - is preserved because v2 is a documented intentional improvement (source-part pattern, Pydantic, insert_selection PK-only return, per-unit part tables, etc ‚Äî see [phase-1b-runtime-regressions.md ¬ß intentional v2 improvements](phase-1b-runtime-regressions.md)),
   - or is an improvement over v1 deliberately taken (B5 multi-segment NWB handling ‚Äî v1 silently mishandled, v2 will fix).

   Phase 1b's scope is therefore much larger than originally planned. The user explicitly elected to keep it as **one phase / one big PR** rather than splitting into 1b+1c, because the items are all v1-parity-themed and benefit from a single review surface. The PR is expected to be substantial; reviewers should expect ~50-item-touching diff plus ~30 new validation tests.

   This decision retroactively justifies the size of Phase 1b. Future phases must NOT use Phase 1b as precedent for absorbing additional scope ‚Äî they remain bound to the original phase-boundary discipline. Phase 1b's audit was specifically a **catching up to v1** exercise driven by the discovery that the Phase 1 implementer had silently diverged from v1 in many places; later phases should not have a comparable backlog.

9. **v1 bugs found during the Phase 1b audit are fixed in v2 AND filed as upstream issues.** Two confirmed bugs in v1's spike-sorting code surfaced during the audit and are handled the same way:
   - **`amplitude_thresh_uV` unit-conversion bug** at [spikesorting/utils.py:171-179](../../../../src/spyglass/spikesorting/utils.py#L171-L179): v1 compares raw int16 NWB counts against a field documented as microvolts. For Frank-lab Intan probes (0.195 ¬µV/count) v1's default of 3000 was effectively ~585 ¬µV; for Neuropixels gain=500 (2.34 ¬µV/count) v1's 3000 was effectively ~7020 ¬µV ‚Äî almost no artifacts ever flagged on Neuropixels in v1 production. v2 fixes this with the clearer `amplitude_threshold_uv` field at [v2/artifact.py:571-572](../../../../src/spyglass/spikesorting/v2/artifact.py#L571-L572) by scaling by gains before the comparison. v2's default of 500 ¬µV is kept (close to v1's effective Intan-probe behavior; scientifically reasonable). Upstream issue filed on LorenFrankLab/spyglass.
   - **Heterogeneous-channel-gains silent `gains[0]` pick** at [v1/recording.py:858](../../../../src/spyglass/spikesorting/v1/recording.py#L858): v1's `conversion = np.unique(recording.get_channel_gains())[0] * 1e-6` silently uses the first channel's gain as the universal conversion factor on the ElectricalSeries, scaling channels with heterogeneous gains by the wrong factor. v2 raises `ValueError` at [v2/recording.py:1100-1109](../../../../src/spyglass/spikesorting/v2/recording.py#L1100-L1109). v2's stricter behavior is kept. Upstream issue filed.

   The pattern: v2 fixes the bug, the v1 bug is documented upstream for the v1 maintainers to address, and the v2 CHANGELOG entry explains the bug + how users with custom v1 thresholds should translate them to v2-equivalent values. This is the precedent for any future v1 bugs surfaced during v2 development ‚Äî fix in v2, file upstream, document the translation. Phase 1b is the first time this pattern was applied; subsequent phases follow it.

## Remaining Open Questions

These are smaller, mostly orthogonal decisions where the plan's default is documented but the user can override.

1. **Pydantic model versioning?** *Current best-answer*: each Lookup table includes a `params_schema_version: int`-style secondary attribute. For production/frozen rows, a Pydantic model shape change increments the schema version and the validator dispatches by version. During authorized pre-production direct-edit windows, the owning phase may keep a version pinned when there is no migration/deprecation surface, but it must state that explicitly and test the validated dump. Documented in [shared-contracts.md](shared-contracts.md).
2. **Sorter container support (Docker / Singularity) in v2?** *RESOLVED*: make it first-class before the user-facing MS4 recommendation lands. Local/Docker/Singularity execution is tracked on `SorterParameters.execution_params`, routed through `_sorting_dispatch.run_si_sorter`, and guarded by preflight. Container flags are not ad hoc sorter params, not `job_kwargs`, and not untracked `run_v2_pipeline` overrides. Recommended container presets must pin container-side SpikeInterface provenance, and MATLAB-backed legacy sorters require explicit tracked container execution rather than the old implicit name-based Singularity fallback.
3. **Curation label enum enforcement?** *RESOLVED (review-fix T4).* `curation_label` is `varchar(32)` (kept varchar for open-ended lab labels ‚Äî NOT a MySQL enum), validated against the canonical `CurationLabel` set (`accept`, `mua`, `noise`, `artifact`, `reject`) on **ALL** insert paths, including direct `CurationV2.UnitLabel.insert1` ‚Äî raw inserts no longer remain free-form. Labels outside the canonical set are rejected unless the caller passes `allow_custom_labels=True` (the explicit escape hatch). (The earlier "DataJoint doesn't support enums" rationale was wrong ‚Äî it does, as `curation_source` shows; the varchar choice is deliberate flexibility, with typo-protection moved to the Python insert boundary.)
4. **Parity baseline SI version mismatch.** *Current best-answer*: v1 parity is no longer performed against minirec (minirec has no real spikes). v1 parity baselines are captured in the legacy SI 0.99 environment when `SPIKESORTING_V2_REAL_NWB_PATH` is set; v2 runs under SI 0.104 and compares against the captured baseline artifacts. Tolerances: `clusterless_thresholder` exact within ¬±1 sample; stochastic sorters within ¬±50% unit count + ¬±30% median firing rate. If the legacy baseline is unavailable, the parity test skips with an explicit message ‚Äî it does not silently relax to a meaningless tolerance.
