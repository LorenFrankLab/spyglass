# Feature parity with v1

[← back to PLAN.md](PLAN.md) · [v1-v2-divergences.md](v1-v2-divergences.md) — per-behavior catalog of intentional v1→v2 departures

This page is the parity contract. It distinguishes behavior v2 must preserve from intentional table/API departures. When in doubt, implementers should preserve downstream behavior and user workflows, not v1's exact table names. For the detailed, code-level catalog of where and why v2 diverges from v1 (with v1 file:line anchors), see [v1-v2-divergences.md](v1-v2-divergences.md).

Runtime compatibility boundary: v2 is the supported runtime path under the
SpikeInterface 0.104 environment introduced by Phase 0c. Existing v0/v1 rows
remain queryable where possible, but active v0/v1 population, MetricCuration,
BurstPair, FigURL, and recompute workflows require the legacy SI 0.99
environment unless Phase 0c explicitly ports a narrow compatibility shim. The
parity targets below describe user-facing workflow successors in v2, not a
promise that v1 runtime code is modernized under SI 0.104.

## Explicit non-parity

- v2 does not promise identical table names or primary keys.
- v2 does not preserve v1's list-returning `insert_selection()` behavior.
- v2 does not auto-create default rows for every installed SpikeInterface sorter.
- v2 does not migrate existing v1 curation rows into `CurationV2`.
- v2 does not guarantee active v0/v1 runtime workflows under the SI 0.104 environment. Legacy runtime workflows require the SI 0.99 environment unless explicitly ported by Phase 0c.
- v2 does not implement `SpikeSortingRecording.update_ids()`. That v1 helper was a transition/backfill tool for recompute-era NWB files; v2's zero-migration policy requires final row fields when each table is introduced.
- v2's boundary-spike handling splits across two paths. The **write** path preserves v1 behavior: `Sorting.make()` runs SpikeInterface's `remove_excess_spikes()` so true boundary-excess spikes are dropped before persistence (guarded by the Phase 1 boundary invariant test). The **readback** path reimplements a `spike_times_to_valid_samples()` helper (`_signal_math.py`) that, when reconstructing frames from the persisted ABSOLUTE spike times, CLAMPS a frame that `searchsorted` lands one past the end back to the last sample if the spike is within ~2 sample periods of the end (floating-point rounding of the final spike's absolute time) — it does NOT drop it, because dropping would desync per-spike features (the `UnitWaveformFeatures` n_feat==n_spikes check). A spike genuinely beyond that tolerance RAISES (an alignment/units bug the write-path drop should have prevented). This readback clamp is a deliberate deviation from v1's drop-only helper.
- v2 does not remove, deprecate, or rewrite v0/v1.
